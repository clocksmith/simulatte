# Simulatte Plugin API v1

**Status:** implemented first-party plugin API  
**Runtime:** browser JavaScript, SDK version `1`  
**Package root:** `public/shared/plugins/<plugin-id>/`

## 1. Scope and trust model

A Simulatte plugin is a governed JavaScript package selected by an experience profile and activated by the shared plugin runtime.

Plugins run in the same JavaScript realm as the host. The boundary is based on:

- a validated manifest;
- least-authority SDK ports;
- immutable inputs and plugin state;
- declared datasets and capabilities;
- integrity-locked code and resources;
- validated request, UI, and presentation contributions;
- static boundary checks.

This is **not** a security sandbox. Arbitrary third-party code requires an iframe, worker isolate, or equivalent boundary.

Durable plugin state should flow through `sdk.events` and `sdk.state`. Auditable claims should be recorded through `sdk.receipts`.

## 2. Package layout

```text
public/shared/plugins/example-plugin/
  plugin.json
  config.schema.json
  default-config.json
  index.js
  helper.js                 # optional
```

`index.js` exports a UMD-compatible factory and exposes the global named by `entry.globalFactory`.

## 3. Manifest contract

Every manifest has the exact schema `simulatte.pluginManifest.v1`. All top-level fields are required, including arrays that are empty.

```json
{
  "schema": "simulatte.pluginManifest.v1",
  "id": "example-plugin",
  "version": "1.0.0",
  "sdkVersion": 1,

  "entry": {
    "path": "./index.js",
    "integrity": "sha384-...",
    "globalFactory": "SimulattePluginExample"
  },

  "resources": [
    { "path": "./config.schema.json", "integrity": "sha384-..." },
    { "path": "./default-config.json", "integrity": "sha384-..." },
    { "path": "./helper.js", "integrity": "sha384-..." }
  ],

  "permissions": [
    "events.propose.v1",
    "receipts.append.v1",
    "state.reduce.v1",
    "ui.inspector.v1"
  ],

  "datasets": [
    { "id": "example.dataset.v1", "required": false }
  ],

  "provides": ["example.capability.v1"],

  "consumes": [
    { "id": "other.capability.v1", "required": false }
  ],

  "extensionPoints": ["event", "ui"],

  "receiptSchemas": [
    "simulatte.plugin.exampleReceipt.v1"
  ],

  "configSchema": "./config.schema.json",
  "defaultConfig": "./default-config.json"
}
```

### 3.1 Field rules

| Field | Rule |
|---|---|
| `id` | Lowercase kebab case: `^[a-z0-9]+(?:-[a-z0-9]+)*$` |
| `version` | Numeric `major.minor.patch` |
| `sdkVersion` | Must equal `1` |
| `entry` | Exact keys: `path`, `integrity`, `globalFactory` |
| `integrity` | Lowercase SHA-384: `sha384-` plus 96 hex characters |
| `resources` | Unique paths; must include `configSchema` and `defaultConfig` |
| `permissions` | Unique values from the SDK v1 permission list |
| `datasets` | `{ id, required }`, optionally with a governed `reference` |
| `provides` | Unique capability IDs |
| `consumes` | Unique `{ id, required }` declarations |
| `extensionPoints` | Any subset of `request`, `route`, `event`, `settlement`, `ui`, `presentation` |
| `receiptSchemas` | Unique receipt schema IDs accepted by `sdk.receipts.append()` |

A dataset reference has the exact shape:

```json
{
  "id": "example.dataset.v1",
  "required": true,
  "reference": {
    "id": "example.dataset.v1",
    "path": "./data/example.json",
    "sha256": "<64-lowercase-hex>",
    "schemaId": "example.datasetSchema.v1"
  }
}
```

The declaration ID and reference ID must match.

### 3.2 Integrity model

- Plugin entry code and declared resources use SHA-384.
- Referenced JSON datasets use SHA-256 plus a schema ID.
- `plugin.json` is validated and embedded in the generated registry; it is not separately declared as a SHA-384 resource.
- Integrity values are generated by the registry sync tool. Do not hand-author final hashes.

## 4. Build and registration

Run:

```bash
npm run plugins:sync
npm run plugins:check
npm test
```

`plugins:sync`:

1. discovers plugin directories;
2. verifies that directory name equals manifest ID;
3. computes entry and resource SHA-384 values;
4. rewrites stale manifest hashes;
5. regenerates `generated-plugin-registry.js`;
6. regenerates plugin script tags in `public/index.html`.

`plugins:check` verifies registry freshness and rejects direct plugin use of:

- `fetch`;
- browser storage APIs;
- DOM APIs;
- host implementation imports;
- cross-plugin imports.

## 5. Runtime lifecycle

For each selected experience, the host:

1. validates the experience profile;
2. resolves plugin manifests and selected configs;
3. validates manifests;
4. builds the capability graph;
5. rejects duplicate providers, missing required providers, self-dependencies, and cycles;
6. verifies plugin entry and resource integrity;
7. creates a dataset-restricted, permission-restricted SDK;
8. activates providers before consumers in deterministic order;
9. validates the returned plugin instances;
10. runs request, route, state, UI, presentation, and settlement hooks;
11. forwards scenario changes through `setScenario`;
12. disposes plugins in reverse activation order.

Inputs passed to plugins are cloned and frozen.

## 6. Factory and plugin instance

### 6.1 UMD factory

```javascript
(function attachExamplePlugin(root, factory) {
  const api = factory();

  if (typeof module === 'object' && module.exports) {
    module.exports = api;
  }

  root.SimulattePluginExample = api;
})(
  typeof globalThis !== 'undefined' ? globalThis : window,
  function createExamplePlugin() {
    function reduce(state, event) {
      if (event.kind === 'example-plugin.incremented') {
        return { ...state, count: state.count + event.amount };
      }
      return state;
    }

    async function activate({ sdk, config, profile, scenario }) {
      sdk.state.register(reduce, { count: 0 });

      function view() {
        return {
          slot: 'inspector',
          title: 'Example',
          rows: [
            { label: 'Count', value: String(sdk.state.read().count) }
          ],
          actions: [
            { id: 'increment', label: 'Increment' }
          ]
        };
      }

      function handleAction(actionId) {
        if (actionId !== 'increment') {
          return { status: 'refused', reason: 'unknown_action' };
        }

        sdk.events.propose({
          pluginId: sdk.pluginId,
          kind: `${sdk.pluginId}.incremented`,
          amount: 1
        });

        return { status: 'settled' };
      }

      return Object.freeze({
        id: 'example-plugin',
        reduce,
        view,
        handleAction,
        dispose() {}
      });
    }

    return Object.freeze({ activate });
  }
);
```

### 6.2 Instance members

The returned instance must contain `id` equal to the manifest ID.

| Member | Type | Extension declaration |
|---|---|---|
| `contributeRequest(context)` | Function | `request` |
| `createRouteContributor(context)` | Function | `route` |
| `reduce(state, event)` | Function | `event` |
| `settle(context)` | Function | `settlement` |
| `view(context)` | Function | `ui` |
| `present(context)` | Function | `presentation` |
| `handleAction(actionId, context)` | Function | None |
| `setScenario(scenario)` | Function | None |
| `dispose()` | Function | None |
| `capabilities` | Object of functions | None |

Extension enforcement is one-way:

- implementing an extension method without declaring its extension point fails;
- declaring an extension point without returning the method is currently allowed.

Returning `reduce` does not register it. State reduction becomes active only through:

```javascript
sdk.state.register(reduce, initialState);
```

Only one reducer may be registered per plugin.

## 7. Host SDK

Every plugin receives:

```javascript
sdk.schema       // "simulatte.pluginSdk.v1"
sdk.pluginId     // manifest ID
sdk.sdkVersion   // 1
sdk.datasets     // restricted immutable dataset view
```

Additional namespaces exist only when their permission is declared.

| SDK namespace | Permission | Current API |
|---|---|---|
| `sdk.capabilities` | `capabilities.invoke.v1` | `invoke(capabilityId, input)` |
| `sdk.clock` | `clock.read.v1` | `instantForMission(mission)` |
| `sdk.events` | `events.propose.v1` | `propose(event)` |
| `sdk.language` | `language.parse.v1` | `parsePrompt(sourceText)` |
| `sdk.receipts` | `receipts.append.v1` | `append(receipt)` plus current receipt helpers |
| `sdk.routing` | `routing.contribute.v1` | `plan`, `alternatives`, `modeFor`, `policy` |
| `sdk.simulation` | `simulation.run.v1` | `run({ id, mission, routeObjective })` |
| `sdk.state` | `state.reduce.v1` | `register(reducer, initialState)`, `read()` |
| `sdk.ui` | `ui.inspector.v1` | Host UI metadata; currently `{ slot: "inspector" }` |
| `sdk.worldQuery` | `world.query.v1` | `snapshot()`, `model()` |

The current receipt port also exposes:

```javascript
sdk.receipts.createReceiptChain()
sdk.receipts.appendReceiptEntry(chain, payload)
sdk.receipts.sha256Hex(value)
sdk.receipts.verifyReceiptChain(chain, options)
```

The full SDK is deeply frozen.

### 7.1 Datasets

`sdk.datasets` is always present and exposes:

```javascript
sdk.datasets.ids
sdk.datasets.require(id)
sdk.datasets.optional(id)
sdk.datasets.receipt(id)
```

Rules:

- only manifest-declared IDs are accessible;
- undeclared access fails;
- missing required datasets fail before activation;
- missing optional datasets return `null`;
- dataset values and receipts are immutable.

A plugin factory may expose validators for custom referenced dataset schemas:

```javascript
return Object.freeze({
  activate,
  datasetValidators: {
    'example.datasetSchema.v1': (value, { world, worldSha256 }) => {
      // Throw on failure, return value or true on success.
      return value;
    }
  }
});
```

The validator schema ID must be declared by one of the plugin's dataset references.

### 7.2 State and events

A proposed event must:

- be an object;
- contain `pluginId` equal to `sdk.pluginId`;
- contain a `kind` beginning with `<plugin-id>.`.

Example:

```javascript
sdk.events.propose({
  pluginId: sdk.pluginId,
  kind: `${sdk.pluginId}.selection-changed`,
  selectionId: 'route-2'
});
```

The host assigns the event schema and sequence, records it, applies the registered reducer, then freezes the new state.

### 7.3 Receipts

```javascript
sdk.receipts.append({
  schema: 'simulatte.plugin.exampleReceipt.v1',
  result: 'accepted'
});
```

The receipt schema must appear in `manifest.receiptSchemas`. The host wraps accepted receipts in a sequenced plugin receipt envelope.

### 7.4 Capabilities

A provider exposes functions using IDs declared in `provides`:

```javascript
return Object.freeze({
  id: 'example-plugin',
  capabilities: {
    'example.capability.v1': (input) => ({ accepted: true, input })
  }
});
```

A consumer must declare the capability in `consumes` before invoking it:

```javascript
const result = sdk.capabilities.invoke(
  'example.capability.v1',
  { value: 42 }
);
```

Rules:

- only one active provider may provide a capability ID;
- required missing providers fail before activation;
- optional missing providers are recorded as disabled;
- consuming an undeclared capability fails;
- self-dependencies and dependency cycles fail;
- provider inputs are cloned and frozen;
- a declared provider function is checked when invoked.

## 8. Extension contracts

## 8.1 Request contribution

```javascript
async function contributeRequest({ sourceText, mission }) {
  // Return null when not applicable.
}
```

The application may call this hook twice:

1. before mission compilation, with no mission;
2. after mission compilation, with the compiled mission.

Treat recognition as idempotent. Avoid duplicate events and receipts during the preflight call.

Accepted contribution:

```javascript
{
  recognized: true,

  obligations: [
    {
      id: 'example-plugin:shade-target',
      kind: 'shade_target',
      required: true
    }
  ],

  unresolved: [],

  executableSourceText: 'Optional canonical mission text',

  missionPatch: {
    routeOverride: {
      segmentIds: ['segment-a', 'segment-b'],
      environmentFieldId: 'optional-field-id',
      selectionId: 'selected-route',
      objective: 12.5,
      algorithm: 'example_route_v1'
    }
  }
}
```

Rules:

- allowed keys are exact;
- `recognized`, `obligations`, and `unresolved` are required;
- obligations have exact keys `id`, `kind`, `required`;
- unresolved entries are non-empty strings;
- route override segment IDs must be non-empty and unique;
- `selectionId`, `objective`, and `algorithm` are required;
- `environmentFieldId` is optional;
- duplicate obligation IDs across plugins fail;
- more than one executable-source override fails;
- more than one route override fails.

## 8.2 Route contribution

```javascript
function createRouteContributor({ mission }) {
  return {
    id: 'example-plugin:risk',

    evaluateSegment({ segment, worldModel, mission, policy, tick }) {
      return {
        eligible: true,
        costDimensions: {
          exampleRisk: 1.2
        },
        rejectionReasons: [],
        receipt: {
          schema: 'example.segmentAudit.v1',
          segmentId: segment.id
        }
      };
    },

    evaluateRoute({ route, worldModel }) {
      return {
        schema: 'example.routeAudit.v1',
        segmentCount: route.segmentIds.length
      };
    }
  };
}
```

Rules:

- the contributor may be `null`;
- `id` and `evaluateSegment` are required;
- contributor IDs must be unique across active plugins;
- `eligible` must be boolean;
- each cost dimension must be finite and non-negative;
- `rejectionReasons` must be an array;
- any rejection makes the segment ineligible;
- profile `routeObjective` weights convert named dimensions into route cost;
- undeclared or unweighted dimensions contribute zero weighted cost;
- `evaluateRoute` is optional and is used for selected-route audit output.

The optional `receipt` returned by `evaluateSegment` is route audit data. It is not appended through `sdk.receipts` and is not checked against `manifest.receiptSchemas`.

## 8.3 Settlement

```javascript
async function settle({ journey }) {
  return {
    obligationResults: [
      {
        obligationId: 'example-plugin:shade-target',
        status: 'settled'
      }
    ],
    stateIdentity: 'selection-42',
    losses: []
  };
}
```

The host awaits settlement after the journey receipt is created and attaches all plugin settlement outputs to the journey.

The shape above is the current first-party convention. SDK v1 does **not** generically validate settlement output or restrict `status` values.

## 8.4 Declarative UI

`view(context)` may return:

- `null`;
- one view object;
- an array of view objects.

```javascript
{
  slot: 'inspector',
  title: 'Example Plugin',

  rows: [
    { label: 'Metric', value: '100 units' }
  ],

  fields: [
    {
      id: 'mode',
      label: 'Mode',
      type: 'select',
      value: 'safe',
      options: [
        { value: 'safe', label: 'Safe' },
        { value: 'fast', label: 'Fast' }
      ]
    }
  ],

  actions: [
    {
      id: 'apply',
      label: 'Apply'
    },
    {
      id: 'focus',
      label: 'Focus route',
      command: {
        kind: 'camera.focus',
        targetId: 'selected-route'
      }
    }
  ]
}
```

Supported slots:

```text
inspector
map
hud
```

Supported field types:

```text
text
date
select
number
```

Rules:

- rows have exact keys `label`, `value`;
- fields are optional;
- select fields require non-empty options;
- actions require `id` and `label`;
- the only current command is `{ kind: "camera.focus", targetId }`;
- camera target IDs are plugin-local; the host namespaces them;
- an action without a command calls:

```javascript
handleAction(actionId, {
  mission,
  routeObjective,
  values
});
```

Field values are delivered as strings. The host owns DOM creation, layout, accessibility, focus, and styling.

## 8.5 Presentation

`present(context)` returns `null` or one validated presentation.

### Presentation v1

```javascript
{
  schema: 'simulatte.pluginPresentation.v1',
  markers: [],
  paths: [],
  actors: [],
  cameraTargets: []
}
```

### Presentation v2

```javascript
{
  schema: 'simulatte.pluginPresentation.v2',
  markers: [],
  paths: [],
  actors: [],
  areas: [],
  sun: {
    id: 'modeled-sun',
    label: 'Modeled sun',
    azimuthDegrees: 180,
    elevationDegrees: 45,
    anchorSegmentIds: ['segment-a'],
    distanceM: 300,
    radiusM: 15,
    intensity: 1
  },
  cameraTargets: []
}
```

v2 requires both `areas` and a non-null `sun` object. Only one active plugin may provide solar lighting.

### Presentation limits

| Contribution | Maximum | Important constraints |
|---|---:|---|
| Markers | 128 | Governed `nodeId`; height `0.2..240`; radius `0.2..40` |
| Paths | 64 | `1..4096` unique segment IDs; width `0.2..24` |
| Actors | 64 | `1..4096` segment IDs; speed `0.1..30`; phase `0..100000` |
| Areas | 384 | `3..128` points; height `0..20` |
| Camera targets | 32 | Up to 128 node IDs and 4096 segment IDs; at least one anchor |
| Sun anchors | 4096 | Azimuth `0..360`; elevation `-5..90`; distance `50..2000` |

All intensities are in `0..2`.

Allowed tones:

```text
cyan green amber red magenta violet blue shade muted
```

Allowed actor kinds:

```text
pedestrian bicycle scooter car
```

The host:

- namespaces contribution IDs as `plugin:<plugin-id>:<local-id>`;
- resolves node and segment IDs against the governed world;
- rejects missing world identities;
- compiles camera targets;
- owns WebGPU, camera state, animation, and DOM rendering.

## 9. Validation and failure behavior

The host fails closed on:

- unknown manifest keys;
- unknown permissions or extension points;
- malformed IDs, versions, or integrity values;
- missing config resources;
- missing required datasets;
- undeclared dataset access;
- duplicate capability providers;
- missing required capabilities;
- capability self-dependencies or cycles;
- instance ID mismatch;
- non-function lifecycle methods;
- extension methods without declarations;
- undeclared receipt schemas;
- duplicate obligation or contributor IDs;
- invalid request, UI, route, or presentation contributions;
- invalid or missing presentation world identities.

Plugin errors are not silently converted into success.

## 10. Current SDK v1 caveats

1. **In-realm execution:** contract and permission enforcement is not a JavaScript isolate.
2. **One-way extension enforcement:** a method requires a declaration; a declaration does not require the method.
3. **Reducer registration:** returning `reduce` does not register it.
4. **Config validation:** `config.schema.json` is integrity-locked, but the plugin runtime does not currently execute that JSON Schema during activation.
5. **Settlement validation:** settlement output is frozen and attached, but not generically schema-validated.
6. **Action and capability results:** return values are not generically schema-validated.
7. **Presentation v2:** a sun object is mandatory, and composed output permits only one sun.
8. **Static boundary checks:** they support first-party governance but are not a hostile-code sandbox.

## 11. Author checklist

Before merging a plugin:

- [ ] Plugin directory and manifest IDs match.
- [ ] Manifest contains every required key.
- [ ] Permissions are minimal.
- [ ] Every dataset and consumed capability is declared.
- [ ] Required versus optional dependencies are intentional.
- [ ] Every implemented extension method has a matching declaration.
- [ ] Reducer is registered exactly once.
- [ ] Event kinds are plugin-namespaced.
- [ ] Request preflight is idempotent.
- [ ] Receipt schemas are declared.
- [ ] Route costs are finite and non-negative.
- [ ] UI and presentation reference only governed IDs.
- [ ] Scenario changes rebuild deterministic state where needed.
- [ ] `dispose()` releases plugin-owned resources.
- [ ] `npm run plugins:sync` has been run.
- [ ] `npm run plugins:check` passes.
- [ ] Relevant tests pass.
